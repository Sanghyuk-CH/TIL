# HTTP reqeust에는 어떤 것들이 있나요? 리스트업 해 주시고 설명해 주세요

GET POST PUT PATCH DELETE OPTIONS
GET 데이터를 읽거나(Read) 검색(Retrieve)할 때에 사용
POST 요청된 리소스 생성
PUT 요청된 리소스 수정(전체데이터)
PATCH 요청된 리소스 수정(일부데이터)
DELETE 요청된 리소스 삭제요청(대부분 안전성 문제로 사용안함)
OPTIONS 웹서버에서 지원되는 메소드의 종류를 확인할 경우 사용

# 브라우저에 URL을 입력하고 요청한 페이지를 볼때까지 어떤 일이 일어나는가?

1. 웹브라우저가 url을 해석
2. URL이 문법에 맞으면 Punycode 인코딩을 url의 호스트 부분에 적용시킨다.
3. HSTS (HTTP Strict Transport Security) 목록을 로드해서 확인한다.
   (HSTS 목록에 있으면 첫 요청을 HTTPS로 보내고 아니면 HTTP로 보낸다.)
4. DNS 조회한다.
5. ARP를 통하여 대상의 IP와 MAC 주소를 알아냄
6. 대상과 TCP 통신을 통해 소켓을 연다.
7. HTTPS의 경우 TLS 핸드 쉐이크 과정이 추가된다.
8. HTTP 프로토콜 요청 (Request)
9. HTTP 서버가 응답한다.

# HTTP와 HTTPS 프로토콜의 차이점은 무엇인가?

하이퍼 텍스트 트랜스퍼 프로토콜의 약자로 W3 상에서 정보를 주고받을 수 있는 프로토콜이고
http에 보안성을 추가하기 위해 SSL을 사용하여 인증서 방식의 암호화 처리를 하여 secure를 추가한것이 https 입니다.

# CORS란 무엇인가요? CORS의 목적과 CORS 활용에 대해 이야기해 주세요

Cross Origin Resource sharing으로 도메인 이름이 서로 다른 사이트간에 api요청을 할 때, 공유를 설정해 주지 않으면 요청을 막는것입니다.
Cors 처리 방법으로 보통 요청을 받는측에서 처리를 해야합니다.
express cors미들웨어를 사용하여 origin method credential의 설정을 해줍니다.

# https 보안 처리 과정

ssl 키를 이용하여 암호화 처리를 합니다.

1. 서버는 클라이언트가 연결해 오면 공개키와 디지털 서명을 포함한 디지털 증명서를 반환합니다.
2. 클라이언트는 디지털 서명을 보고 디지털 증명서가 올바른 것인지 체크하고, 올바른 증명서라면 공통 키의 재료를 공개키로 암호화하여 서버에게 전달합니다.
3. 서버로 보내져 온 데이터를 자신의 비밀키로 복호화하고 공통키의 재료를 꺼냅니다. 1 ~ 3에 대한 과정을 SSL handshake라고 부릅니다.
4. 클라이언트와 서버는 공통 키의 재료로부터 공통 키를 만들어 이후에는 그것을 이용하여 암호화 통신을 수행합니다.
